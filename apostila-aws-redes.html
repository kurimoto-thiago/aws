1.1 — O que é Computação em Nuvem?

Computação em nuvem é o fornecimento de recursos computacionais — servidores, armazenamento, banco de dados, redes, software — via Internet, com pagamento pelo uso. Em vez de comprar e manter hardware físico, você aluga capacidade de grandes datacenters.

🏗️
Analogia de RedesVocê já conhece a diferença entre ter um servidor físico na escola (on-premises) e acessar um sistema via VPN remota. Nuvem é como ter toda a infraestrutura de redes da escola hospedada remotamente — sem precisar gerenciar o hardware físico, a energia, o ar-condicionado ou a manutenção.

1.3 — Modelos de Implantação

  • Nuvem pública: Recursos compartilhados da AWS. Mais barato, escala imediata.
  • Nuvem privada: Infraestrutura dedicada (AWS Outposts). Controle total, custo alto.
  • Nuvem híbrida: Parte on-premises, parte na AWS. Transição gradual ou requisitos legais.
  • Multi-cloud: AWS + Azure + GCP. Evita lock-in, mais complexo.

1.4 — Benefícios da Nuvem

  • Escalabilidade: Sobe e desce recursos em minutos, não meses.
  • Elasticidade: Ajuste automático à demanda (Black Friday, por exemplo).
  • Pay-as-you-go: Pague apenas o que usar, pelo tempo que usar.
  • Alta disponibilidade: Múltiplos datacenters redundantes.
  • Segurança: Conformidade com ISO 27001, SOC 2, PCI-DSS, LGPD.
  • Velocidade de inovação: Deploy em horas, não semanas.

3.1 — O que é EC2?

O Amazon EC2 (Elastic Compute Cloud) oferece servidores virtuais na nuvem chamados instâncias. Você escolhe o sistema operacional, o hardware virtual (vCPUs, RAM, disco) e a localização geográfica. Em minutos, tem um servidor pronto.

🖥️
Conexão com RedesEC2 é como a VM do VirtualBox que você configurou nas aulas de redes — mas hospedada em datacenter da AWS, com IP público, acessível via SSH de qualquer lugar, e com uptime de 99.99%.

4.1 — VPC: Sua Rede Privada na AWS

A Amazon VPC (Virtual Private Cloud) é uma rede virtual isolada dentro da AWS. Pense nela como seu próprio datacenter lógico dentro da nuvem. Você controla: range de IPs (CIDR), subnets, tabelas de roteamento, gateways e ACLs.

🌐
Conexão Direta com RedesVPC é a implementação em nuvem do que você já estudou: VLAN, subnet, roteador, firewall, NAT. O CIDR da VPC é como o endereçamento de uma rede corporativa. As subnets públicas têm Internet Gateway (como interface WAN). As subnets privadas usam NAT Gateway (como NAT do roteador).

Criar VPC via AWS CLI
💻 Terminal
Terminal — Criar VPC passo a passo
# PASSO 1: Criar a VPC
aws ec2 create-vpc \
  --cidr-block 10.0.0.0/16 \
  --tag-specifications 'ResourceType=vpc,Tags=[{Key=Name,Value=vpc-redes-cli}]' \
  --region sa-east-1
{"Vpc": {"VpcId": "vpc-0abc123", "CidrBlock": "10.0.0.0/16", "State": "available"}}

# Salvar VPC ID em variável
VPC_ID=vpc-0abc123

# PASSO 2: Criar subnet pública
aws ec2 create-subnet \
  --vpc-id $VPC_ID \
  --cidr-block 10.0.1.0/24 \
  --availability-zone sa-east-1a \
  --tag-specifications 'ResourceType=subnet,Tags=[{Key=Name,Value=subnet-publica-1a}]'
{"Subnet": {"SubnetId": "subnet-0def456", "CidrBlock": "10.0.1.0/24"}}

# PASSO 3: Criar Internet Gateway e anexar à VPC
IGW_ID=$(aws ec2 create-internet-gateway --query "InternetGateway.InternetGatewayId" --output text)
aws ec2 attach-internet-gateway --internet-gateway-id $IGW_ID --vpc-id $VPC_ID

# PASSO 4: Configurar route table para subnet pública
RT_ID=$(aws ec2 create-route-table --vpc-id $VPC_ID --query "RouteTable.RouteTableId" --output text)
aws ec2 create-route --route-table-id $RT_ID --destination-cidr-block 0.0.0.0/0 --gateway-id $IGW_ID
aws ec2 associate-route-table --route-table-id $RT_ID --subnet-id subnet-0def456

✓ VPC com subnet pública e Internet Gateway criados via CLI!

5.1 — Application Load Balancer (ALB)

O ALB distribui o tráfego HTTP/HTTPS entre múltiplas instâncias EC2. Funciona na camada 7 (Aplicação) do modelo OSI, permitindo roteamento inteligente por URL, headers ou cookies.

1
Criar Application Load Balancer
🖥️ Console + 💻 CLI
Via Console:
1. EC2 → Load Balancers → Criar load balancer
2. Tipo: Application Load Balancer
3. Nome: alb-redes-escola
4. Scheme: voltado para a Internet
5. VPC: selecione a VPC criada no Cap 04
6. Mapeamentos: selecione as 2 subnets públicas
7. Security Group: permitir HTTP porta 80
8. Target Group: criar novo → tipo Instâncias → porta 80 → HTTP
9. Registrar as 2 instâncias EC2 no target group
10. Criar e aguardar status Active
Terminal — Criar ALB via CLI
# Criar Target Group
aws elbv2 create-target-group \
  --name     tg-redes-escola \
  --protocol HTTP \
  --port     80 \
  --vpc-id   vpc-0abc123 \
  --health-check-path /
{"TargetGroups": [{"TargetGroupArn": "arn:aws:elasticloadbalancing:..."}]}

# Registrar instâncias no Target Group
aws elbv2 register-targets \
  --target-group-arn arn:aws:elasticloadbalancing:... \
  --targets Id=i-0abc123 Id=i-0def456

# Criar ALB
aws elbv2 create-load-balancer \
  --name    alb-redes-escola \
  --subnets subnet-0def456 subnet-0ghi789 \
  --security-groups sg-0jkl012
{"LoadBalancers": [{"DNSName": "alb-redes-escola-123.sa-east-1.elb.amazonaws.com"}]}
✓ Acesse o site pelo DNS do ALB!

8.1 — O que são Edge Locations?

Edge Locations são pontos de presença (PoPs) da AWS distribuídos pelo mundo. Atualmente são mais de 600 em 90 cidades. O CloudFront usa essas Edge Locations para entregar conteúdo ao usuário final a partir do ponto mais próximo geograficamente.

🌐
CDN e Protocolos de RedeUma CDN funciona como um sistema de cache distribuído. Quando um usuário em Belém acessa seu site hospedado em São Paulo, sem CDN a requisição percorre ~2500km. Com CloudFront, o conteúdo é cacheado em uma Edge Location em Belém, e a latência cai de 150ms para 5ms. É o mesmo princípio do DNS cache que você estudou em redes.

9.1 — O que é Serverless?

Serverless não significa que não há servidor — significa que você não gerencia o servidor. O código roda em resposta a eventos, escala automaticamente e você paga apenas pelo tempo de execução (em milissegundos).

⚡
Eventos em RedesVocê já conhece eventos de rede: pacote chega na interface → processa → encaminha. Lambda funciona igual: evento chega (upload S3, requisição HTTP, mensagem SQS) → código executa → resultado retorna. É programação orientada a eventos no nível de infraestrutura.

Código Python — Lambda Function
# Código da função Lambda — cole no editor do Console
import json, boto3, os

def lambda_handler(event, context):
    # Extrair informações do evento
    s3 = boto3.client('s3')
    
    if 'Records' in event:
        # Evento do S3 — arquivo foi enviado
        bucket = event['Records'][0]['s3']['bucket']['name']
        arquivo = event['Records'][0]['s3']['object']['key']
        tamanho = event['Records'][0]['s3']['object']['size']
        
        print(f"Arquivo recebido: {arquivo}")
        print(f"Bucket: {bucket} | Tamanho: {tamanho} bytes")
        
        return {
            'statusCode': 200,
            'body': json.dumps({
                'mensagem': f'Arquivo {arquivo} processado!',
                'tamanho_bytes': tamanho
            })
        }
    
    return {'statusCode': 200, 'body': 'Função executada sem evento S3'}
Terminal — Lambda via CLI
# Empacotar código para deploy
zip function.zip lambda_function.py

# Criar função Lambda via CLI
aws lambda create-function \
  --function-name  processa-upload-cli \
  --runtime        python3.12 \
  --handler        lambda_function.lambda_handler \
  --role           arn:aws:iam::123456789012:role/lambda-basic-role \
  --zip-file       fileb://function.zip

# Testar a função
aws lambda invoke \
  --function-name processa-upload-cli \
  --payload       '{"test": "hello"}' \
  output.json
cat output.json
{"statusCode": 200, "body": "Função executada sem evento S3"}

from http.server import HTTPServer, BaseHTTPRequestHandler
import json, socket

class Handler(BaseHTTPRequestHandler):

        self.send_response(200)
        self.send_header("Content-type", "application/json")
        self.end_headers()

HTTPServer(("", 8080), Handler).serve_forever()
